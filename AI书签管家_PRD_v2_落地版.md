## 1. 问题陈述

### 1.1 核心问题

v1 已实现 AI 对话找书签、一键收藏+自动分类、悬浮球入口、批量管理等 P0 功能，但用户真实反馈是 **"很多功能不好用"**。根因不在功能做错了，而在 **价值主张停留在"管理工具"心智**——用户要的不是"把书签整理好"，而是 **"我收藏过的东西不要安静积灰，在合适的时候重新出现在我面前，形成我的知识脉络"**。

### 1.2 问题量化

| 维度 | v1 现状 | 问题根因 |
|------|---------|----------|
| 7 日留存 | ~15% | 安装后无"离不开"的 Aha 时刻，用户逐渐遗忘 |
| 周活跃率 | ~25% | 仅在收藏/查找时使用，无日常打开理由 |
| 用户平均书签数 | 50+ | 用户"不敢收藏更多"——因为越多越乱 |
| AI 功能周使用频次 | ~3 次 | AI 价值仅在"搜一下"，缺少持续价值输出 |
| 用户主动推荐率 | < 5% | 无可分享的成果，社交传播缺位 |

### 1.3 不解决的成本

- **用户流失**：安装 → 偶尔用 → 卸载的循环无法打破，Chrome Web Store 评分下降
- **竞争风险**：Raindrop.io 持续迭代 AI 能力，Chrome 自身也在集成 AI 功能，窗口期 6-12 个月
- **商业化受阻**：纯"管理书签"价值有限，难以构建付费分层

### 1.4 v2 核心命题

> **让书签从"存档"变成"可被重新发现的知识资产"。**

所有新功能只回答三个问题：

1. **怎么看？** — 多维视图，从不同角度重新理解收藏
2. **怎么再发现？** — 旧书签主动浮现，不等用户翻文件夹
3. **怎么增值？** — 孤立书签产生联系，从 URL 列表变成可思考的知识

## 2. 目标与非目标

### 2.1 用户目标

| # | 目标 | 衡量方式 |
|---|------|----------|
| U1 | 用户每天有理由打开扩展查看"再发现"推送 | 日活率提升至 40%+ |
| U2 | 用户能用多维视图（时间轴/主题/图谱）重新理解收藏 | 视图切换深度 ≥ 3 |
| U3 | 用户能一键清理失效书签和重复书签 | 失效清理转化率 > 40% |
| U4 | 用户收藏书签时能附带上下文（备注/标签/高亮） | 带备注书签占比 > 30% |

### 2.2 业务目标

| # | 目标 | 衡量方式 |
|---|------|----------|
| B1 | 7 日留存率从 ~15% 提升至 40% | 安装后第 7 天活跃用户占比 |
| B2 | 周活跃率从 ~25% 提升至 60% | WAU / 安装量 |
| B3 | AI 功能周使用频次从 ~3 次提升至 10+ 次 | 搜索 + 再发现 + 主题交互合计 |
| B4 | 用户主动推荐率从 < 5% 提升至 15%+ | 安装来源追踪 + 分享按钮点击 |

### 2.3 非目标（Non-Goals）

| # | 非目标 | 原因 |
|---|--------|------|
| NG1 | 不做浏览器 Tab Session 管理 | 那是 Workona/Toby 的赛道，我们聚焦书签本身 |
| NG2 | 不做完整 RSS 阅读器 | 超出书签管理范畴，属于 Feedly/Inoreader 赛道 |
| NG3 | 不做笔记主体功能 | 做"书签 + 轻笔记"，不做 Notion 替代品 |
| NG4 | 不做团队空间/协作 | 早期聚焦个人场景，团队协作需后端服务和账号体系 |
| NG5 | 不做跨浏览器支持（Firefox/Safari） | 先在 Chrome 验证 v2 PMF |
| NG6 | v2 首发不做跨设备云同步 | 云同步需后端基建，放在 P2 阶段 |
| NG7 | 不做付费订阅 | v2 依赖用户自配 API Key，不涉及平台计费 |

## 3. 用户画像

### 画像 A：信息囤积型用户（占比 ~45%）

| 维度 | 描述 |
|------|------|
| 特征 | 收藏数百个书签但很少整理，"先收藏再说"心态 |
| 痛点 | 书签太多找不到，想整理但不知从何下手，越攒越多 |
| 核心需求 | 一键自动分类 + 批量清理 + 失效链接检测 |
| v2 触发点 | "再发现"推送让它想起老书签；"失效清理"一键解决堆积 |

### 画像 B：效率型知识工作者（占比 ~35%）

| 维度 | 描述 |
|------|------|
| 特征 | 经常收藏技术文档、研究资料，有分类意识但维护成本高 |
| 痛点 | 记得"收藏过什么内容"但记不住标题；一个书签属于多个主题但文件夹只能放一个 |
| 核心需求 | AI 语义搜索 + 多标签系统 + 主题视图 |
| v2 触发点 | 主题视图把散落资料自动聚成主题；知识图谱发现隐藏关联 |

### 画像 C：重度 AI 用户（占比 ~20%）

| 维度 | 描述 |
|------|------|
| 特征 | 已有 OpenAI/Claude API Key，习惯用 AI 辅助日常任务 |
| 痛点 | 现有书签工具不支持 AI，想要 AI 深度参与知识管理 |
| 核心需求 | 可配置模型 + AI 摘要 + 知识图谱 + 画布视图 |
| v2 触发点 | AI 摘要让"稍后读"变为"已消化"；画布视图让知识可视化 |

## 4. 用户故事

### 4.1 P0 用户故事

**US-V2-01：失效链接检测与一键清理**
> 作为信息囤积型用户，我希望扩展自动检测哪些书签链接已经失效（404/5xx/重定向），并提供一键清理功能，以便我快速清理无效书签，保持书签库健康。

**US-V2-02：重复书签检测**
> 作为经常收藏页面的用户，我希望扩展能帮我找出重复的书签（相同 URL 或相同标题），以便我合并或删除冗余条目，减少混乱。

**US-V2-03：每日再发现推送**
> 作为所有类型的用户，我希望每天能收到 3-5 条"你可能忘了的"老书签推送，以便我在合适的时机重新发现曾经收藏的有价值内容，而不是让它们安静积灰。

**US-V2-04：时间轴视图**
> 作为所有类型的用户，我希望能按时间顺序查看我的书签收藏历史（今天/昨天/上周/本月/更早），以便我对"我的数字积累"产生具象感知，回顾自己的知识沉淀轨迹。

**US-V2-05：多标签系统**
> 作为效率型知识工作者，我希望能给一个书签打多个标签（而非只能放一个文件夹），以便一个书签能从多个维度被找到——比如一篇 React 文章可以同时属于 `tech/frontend/react` 和 `tech/performance`。

**US-V2-06：收藏时备注与高亮**
> 作为所有类型的用户，我希望在收藏书签时能添加一段备注（"为什么收藏这个"）和标记关键内容高亮，以便半年后回看时能想起收藏的上下文。

**US-V2-07：视图切换导航**
> 作为所有类型的用户，我希望能用顶部 Tab 在不同视图（对话/列表/时间轴）之间快速切换，并用键盘快捷键（Cmd/Ctrl + 1~6）直达，以便我高效地在不同视角间探索。

### 4.2 P1 用户故事

**US-V2-08：主题视图（AI 聚类）**
> 作为效率型知识工作者，我希望 AI 自动把散落的书签聚成主题（如"React 进阶""设计灵感""AI 资讯"），以便我从"主题"维度理解我的知识结构，而非只能按文件夹浏览。

**US-V2-09：知识图谱视图**
> 作为效率型知识工作者，我希望能看到一个力导向图谱，节点是书签、边是关联（共享标签/AI 识别的语义相似），以便我发现"看似无关"的书签之间的隐藏联系。

**US-V2-10：稍后读队列**
> 作为信息囤积型用户，我希望能把"想详细阅读"的书签加入"稍后读"队列，AI 帮我生成摘要，以便我不用全文阅读就能判断是否值得深入。

**US-V2-11：每日/每周回顾推送**
> 作为所有类型的用户，我希望每天/每周收到回顾推送（"本周你收藏了 N 个书签，其中 M 个关于 X 主题"），以便我保持对知识积累的感知。

### 4.3 P2 用户故事

**US-V2-12：画布视图**
> 作为效率型知识工作者，我希望能在无限画布上自由摆放书签卡片、连线、加文字笔记，以便我像用 Notion 一样组织我的知识地图，并一键导出为图片分享。

**US-V2-13：跨设备同步**
> 作为所有类型的用户，我希望在公司的收藏能同步到家中的电脑，以便我的知识库不被设备割裂。

## 5. 功能需求 — P0 详设

### 5.1 失效链接检测 + 一键清理（F1）

#### 5.1.1 功能描述

扩展后台定期扫描所有书签 URL，检测失效状态（404/5xx/重定向到首页/内容已删/DNS 失败），在面板"清理"入口展示结果，支持一键批量清理。

#### 5.1.2 验收标准

**扫描机制**
- [ ] 安装后首次打开面板，自动触发一次后台扫描（不阻塞 UI）
- [ ] 后续每天凌晨 3:00 自动扫描（通过 `chrome.alarms` 定时）
- [ ] 用户可在设置中手动触发"立即检测"
- [ ] 检测方式：HEAD 请求，超时 10 秒
- [ ] 并发检测数限制为 5，避免网络拥塞
- [ ] 检测结果缓存到 `chrome.storage.local`，TTL 24 小时
- [ ] 扫描进度在面板中以进度条显示（已检测 N / 总数 M）

**失效判定规则**
- [ ] HTTP 404 → 标记为"失效-404"
- [ ] HTTP 5xx → 标记为"失效-服务器错误"
- [ ] HTTP 3xx 重定向到首页或根域名 → 标记为"失效-重定向"
- [ ] DNS 解析失败 → 标记为"失效-DNS"
- [ ] 超时（10 秒无响应）→ 标记为"失效-超时"
- [ ] HTTP 200 但 Content-Length < 100 字节 → 标记为"失效-内容已删"（需进一步验证）

**清理交互**
- [ ] 面板新增"清理"入口（Tab 或按钮）
- [ ] 失效书签按失效类型分组展示（可折叠）
- [ ] 每条显示：favicon、标题、URL（截断）、失效类型标签、检测时间
- [ ] 支持"全选失效"→ 二次确认弹窗 → 批量删除
- [ ] 支持"保留备份到 Archive.org"选项（删除前可选）
- [ ] 删除的书签进入回收站（v1 已有），保留 30 天
- [ ] 检测结果可导出为 CSV（URL、标题、失效类型、检测时间）

**性能约束**
- [ ] 1000 书签全量扫描完成时间 < 5 分钟
- [ ] 扫描过程不影响用户正常浏览（低优先级请求）
- [ ] 失效检测结果列表渲染使用虚拟滚动

**隐私约束**
- [ ] 仅访问用户书签 URL 本身进行 HEAD 请求
- [ ] 不上传书签数据到任何服务器（纯本地检测）
- [ ] 扫描结果仅存储在 `chrome.storage.local`

#### 5.1.3 依赖

- 无外部依赖，纯 Content Script / Service Worker 内完成

#### 5.1.4 数据影响

- 新增 `InvalidLinkRecord` 类型（见数据模型 8.3）
- 新增 `chrome.alarms` 定时任务

### 5.2 重复书签检测（F2）

#### 5.2.1 功能描述

检测完全相同 URL 或相同标题的书签，提示用户合并或删除冗余条目。

#### 5.2.2 验收标准

**检测逻辑**
- [ ] URL 完全相同（忽略尾部斜杠和 query 参数排序差异）→ 标记为"URL 重复"
- [ ] 标题完全相同（忽略大小写和首尾空格）→ 标记为"标题重复"
- [ ] URL 指向同一页面但参数不同（如 `?utm_source=` 等 tracking 参数）→ 标记为"疑似重复"
- [ ] 检测在本地完成，无需网络请求

**展示与操作**
- [ ] 面板"清理"入口下新增"重复书签"子 Tab
- [ ] 重复书签按组展示，每组显示：重复数量、代表书签、所有副本列表
- [ ] 每组提供"保留最新""保留最旧""保留第一个"快捷操作
- [ ] 支持展开查看所有副本的详情（收藏时间、所在文件夹、标签）
- [ ] 用户可选择保留哪个，其余自动移入回收站
- [ ] 提供"一键清理所有重复组"按钮（保留每组最新的）

**边界情况**
- [ ] 同一 URL 出现在 3+ 个文件夹 → 全部列出，用户选择保留策略
- [ ] URL 仅有 tracking 参数不同 → 标注但默认不自动合并（需用户确认）
- [ ] 检测结果不缓存（每次打开"清理"实时计算，O(n) 复杂度可接受）

#### 5.2.3 依赖

- 无外部依赖

### 5.3 再发现 — Daily Resurfacing（F3）⭐

#### 5.3.1 功能描述

每天主动向用户推送 3-5 条"你可能忘了的"老书签，让旧书签主动浮现而非等用户翻找。

#### 5.3.2 验收标准

**触发机制**
- [ ] 用户安装满 7 天后开始推送（确保有足够数据积累）
- [ ] 推送时机：每天用户首次打开浏览器时（监听 `chrome.runtime.onStartup`）+ 每天早上 9:00（`chrome.alarms`，可配置）
- [ ] 推送数量：每次 3-5 条（可配置）
- [ ] 推送渠道：浏览器通知（`chrome.notifications` API）+ 面板内"再发现"入口

**选书算法**
- [ ] 候选池：收藏时间 > 30 天 且 最近访问时间 > 14 天 的书签
- [ ] 评分公式：
  ```
  score = (
    days_since_last_visit * 0.3 +        // 越久没看，分数越高
    min(days_since_added, 365) * 0.2 +    // 越老的书签分越高，但封顶 1 年
    topic_match_score * 0.3 +            // 与用户近期浏览主题相关度
    bookmark_quality_score * 0.2          // 用户评分 / 历史访问频次
  )
  ```
- [ ] 去重：已推送过的书签 30 天内不再推送
- [ ] 用户标记"不感兴趣"的书签 90 天内不再推送
- [ ] 推送结果缓存到 `chrome.storage.local`，同一天内不重复计算

**交互**
- [ ] 浏览器通知：显示书签标题 + "3 个月前收藏"，点击直接打开书签
- [ ] 面板"再发现"入口：显示卡片流，每张卡片含 favicon、标题、域名、收藏时间、AI 标注"这是你 N 天前的收藏"
- [ ] 每张卡片支持：打开、再收藏（重置时间）、归档（不再推送）、删除
- [ ] 一键"换一批"按钮：重新选取 3-5 条不同的书签
- [ ] 卡片流支持无限滚动（每次加载 5 条）

**用户控制**
- [ ] 设置中提供"再发现推送"总开关
- [ ] 可调整推送频率：每天 / 每周 / 关闭
- [ ] 可调整推送时间（默认 09:00）
- [ ] 可调整每次推送数量（3-10 条）
- [ ] "不感兴趣"反馈：卡片上提供 ❌ 按钮，点击后 90 天不再推送该书签

**反打扰机制**
- [ ] 连续 3 次推送用户未点击 → 自动降频为每周推送
- [ ] 连续 5 次推送用户未点击 → 暂停推送，面板内显示提示"已暂停推送，点击恢复"
- [ ] 推送恢复后重置计数器

#### 5.3.3 依赖

- 需要 `chrome.notifications` API 权限（manifest 已有）
- 需要 `chrome.alarms` API 权限（已有）
- 与"主题视图"（F7）有联动：再发现推送可基于主题匹配

#### 5.3.4 数据影响

- 新增 `ResurfaceRecord` 类型（见数据模型 8.4）
- 新增 `chrome.alarms` 定时任务
- 新增 `chrome.notifications` 通知

### 5.4 时间轴视图（F4）

#### 5.4.1 功能描述

按日期分组展示书签收藏历史，让用户对"我的数字积累"产生具象感知。

#### 5.4.2 验收标准

**视图入口**
- [ ] 面板顶部 Tab 栏新增"时间轴"视图切换按钮
- [ ] 键盘快捷键 `Cmd/Ctrl + 3` 直达时间轴视图（1=对话，2=列表，3=时间轴）
- [ ] 视图偏好记忆到 `chrome.storage.local`

**分组展示**
- [ ] 按时间分组：今天 / 昨天 / 上周 / 本月 / 今年 / 更早

## 9. 技术方案与约束

### 9.1 技术栈（延续 v1）

| 模块 | 技术方案 |
|------|----------|
| 扩展框架 | Chrome Extension Manifest V3 |
| 前端框架 | React 18 + TypeScript (strict) |
| UI 组件 | Tailwind CSS + Headless UI |
| 状态管理 | Zustand |
| 书签 API | `chrome.bookmarks.*` |
| 本地存储 | `chrome.storage.local` |
| 定时任务 | `chrome.alarms` API |
| 通知推送 | `chrome.notifications` API |
| AI 调用 | Fetch API → 用户配置的模型 API（SSE 流式） |
| 构建工具 | Vite + @crxjs/vite-plugin |
| 虚拟滚动 | @tanstack/react-virtual |

### 9.2 v2 架构扩展

```
┌─────────────────────────────────────────────────────┐
│              交互层（UI Layer）                       │
│  悬浮面板 · AI 对话 · 列表 · 时间轴 · [主题] · [图谱]  │
│  清理中心(失效+重复) · 再发现卡片流 · 备注编辑器       │
├─────────────────────────────────────────────────────┤
│            核心功能层（Core Functions）                │
│  一键收藏+备注 · 智能分类+标签 · 再发现引擎            │
│  失效检测 · 重复检测 · 标签系统 · 回顾推送             │
├─────────────────────────────────────────────────────┤
│             AI 引擎层（AI Engine）                     │

│  Chrome Bookmarks API · chrome.storage.local         │
│  Tags · Notes · Highlights · InvalidLinks · Resurface│
├─────────────────────────────────────────────────────┤
│            定时任务层（Scheduler）                      │
│  chrome.alarms: 失效扫描(每日) · 再发现推送(每日)     │
│  chrome.notifications: 推送通知                       │
└─────────────────────────────────────────────────────┘
```

### 9.3 关键技术决策

| 决策 | 选择 | 原因 |
|------|------|------|
| AI 搜索策略 | Prompt 注入 + 关键词预筛 | v1 已验证可行，v2 保持；> 500 书签时预筛 |
| 标签存储 | `chrome.storage.local` | 标签独立于 Chrome 文件夹，需自管理 |
| 失效检测方式 | HEAD 请求 | 比 GET 轻量，不下载页面内容 |
| 再发现推送 | `chrome.alarms` + `chrome.notifications` | MV3 原生支持，无需外部服务 |
| 视图切换性能 | 虚拟滚动 + Web Worker | 1000+ 书签分组计算不阻塞 UI |
| 高亮定位 | XPath 存储（best-effort） | 页面变动后可能失效，需降级提示 |

### 9.4 性能约束

| 场景 | 约束 |
|------|------|
| 视图切换（1000 书签） | < 200ms |
| 失效扫描（1000 书签全量） | < 5 分钟 |
| 再发现选书计算 | < 500ms |
| 标签筛选响应 | < 100ms |
| 面板展开动画 | < 300ms |
| AI 搜索响应 | < 5 秒（取决于模型 API 延迟） |

### 9.5 隐私与合规

| 数据 | 存储位置 | 是否上传 | 合规要求 |
|------|----------|----------|----------|
| 书签数据 | Chrome 原生 | 否 | 不涉及 |
| 标签/备注/高亮 | `chrome.storage.local` | 否 | 用户可导出/删除 |
| AI 查询内容 | 仅发送到用户配置的 API | 仅 AI API 调用 | 设置面板需展示隐私声明 |
| 失效检测结果 | `chrome.storage.local` | 否 | 不涉及 |
| 埋点数据 | `chrome.storage.local` | 否（匿名聚合） | 不关联个人身份 |
| API Key | `chrome.storage.local` | 仅 AI API 调用 | 密码模式显示 |

## 12. 风险与对策

| # | 风险 | 等级 | 影响 | 对策 | 负责人 |
|---|------|------|------|------|--------|
| R1 | Chrome 内置 AI 替代核心功能 | 高 | 核心功能被原生替代，用户流失 | 加速走"再发现 + 主题视图"差异化路线，做 Chrome 不做的主动推送 | 产品 |
| R2 | 用户不愿配 API Key | 中 | AI 功能无法使用，价值感知低 | 支持 Ollama 本地模型（零成本）；基础检测功能（失效/重复）不依赖 AI | 工程 |
| R3 | AI 成本失控 | 中 | 用户因费用放弃使用 | 默认本地 embedding + 缓存；AI 调用仅在"创造价值时刻"（分类/摘要/搜索） | 工程 |
| R4 | 知识图谱性能差（P1） | 中 | 200+ 节点卡顿 | 子图渲染 + 懒加载 + 虚拟化；超过 200 节点只渲染活跃子图 | 工程 |
| R5 | 标签系统与文件夹冲突 | 中 | 用户认知混乱，不知道该用哪个 | 提供"标签为主 / 文件夹为主 / 并存"三种模式；首次使用引导 | 产品+设计 |
| R6 | 再发现推送打扰用户 | 中 | 用户卸载扩展 | 反打扰机制（连续未点击自动降频→暂停）；推送频率可配置 | 产品 |
| R7 | `chrome.storage.local` 容量限制 | 低 | 数据量超 10MB 时性能下降 | 数据压缩 + 定期清理过期记录 + 增量存储 | 工程 |
| R8 | Chrome Web Store 审核风险 | 低 | 推送通知权限可能被审核质疑 | 准备权限使用说明文档；通知为可选功能 | 合规 |
