## 1. 产品概述

**AI 书签管家** 是一款 Chrome 浏览器扩展，将 AI 对话能力与书签管理深度融合。用户通过浏览器边缘的**轻量悬浮球**，即可在任何网页上一键展开悬浮面板，用自然语言查找书签、一键收藏当前页面并自动归类、批量清理冗余书签。产品核心价值主张：**"收藏即整理，搜索即对话"**。

### 核心差异化

与 Chrome 原生书签管理器和其他书签扩展相比，本产品的差异在于：

- **AI 对话式交互**：用户不必手动翻找文件夹树，直接用自然语言问"我之前收藏的那个 React 性能优化的文章呢？"，AI 语义检索返回结果
- **一键收藏 + 自动分类**：收藏当前页面时，AI 自动分析页面内容并归入合适的分类，无需用户手动选择文件夹
- **可配置模型**：用户可自行配置 API Key 和模型（OpenAI / Claude / 本地模型），不绑定单一服务商

## 2. 问题陈述

### 2.1 用户痛点

Chrome 书签管理长期存在三个核心问题：

1. **书签坟场效应**：用户收藏的书签数量随时间膨胀，最终变成一个无人维护的"坟场"。据 Chrome 使用数据，活跃用户的平均书签数超过 200 个，其中 40% 以上在收藏后从未被再次访问。
2. **查找困难**：Chrome 原生书签管理器只支持标题/URL 的精确匹配搜索，不支持语义搜索。用户记得"内容是关于什么"但记不住标题时，几乎无法找到。
3. **整理成本高**：手动将书签拖入文件夹、检查重复链接、删除失效书签——这些操作繁琐且没有批量处理能力，用户倾向于"先收藏以后再说"，导致问题持续恶化。

### 2.2 不解决的成本

- 用户因找不到已收藏的书签而重复搜索、重复收藏，浪费大量时间
- 书签列表混乱降低浏览效率，部分用户直接放弃使用书签功能
- 竞品（如 Raindrop.io、Pinboard）已提供 AI 分类能力，Chrome 原生体验差距持续扩大

## 3. 目标

### 3.1 用户目标

| # | 目标 | 衡量方式 |
|---|------|----------|
| U1 | 用户能通过自然语言在 3 秒内找到目标书签 | AI 搜索平均响应时间 < 3s，结果准确率 > 85% |
| U2 | 用户一键收藏时自动完成分类，无需手动选择文件夹 | 自动分类准确率 > 80%，用户手动调整率 < 20% |
| U3 | 用户能高效批量清理冗余书签 | 批量删除/移动操作平均完成时间 < 30s |

### 3.2 业务目标

| # | 目标 | 衡量方式 |
|---|------|----------|
| B1 | 扩展上线后 3 个月内获得 10,000+ 安装用户 | Chrome Web Store 安装量 |
| B2 | 用户留存率（7 日活跃）达到 35%+ | DAU/安装量比 |
| B3 | AI 功能使用率达到 60%+ | 使用过 AI 对话或智能分类的用户占比 |

## 4. 非目标（Non-Goals）

本版本明确不做以下事项：

| # | 非目标 | 原因 |
|---|--------|------|
| NG1 | 不做跨浏览器同步（Firefox / Safari 支持） | 先聚焦 Chrome 生态，验证 PMF 后再扩展 |
| NG2 | 不做书签内容全文索引和离线阅读 | 超出书签管理范畴，属于 Pocket/Readwise 赛道 |
| NG3 | 不做团队协作/共享书签 | 需要后端服务和账号体系，v1 做纯本地+AI API 调用的轻量方案 |
| NG4 | 不做付费订阅模型 | v1 依赖用户自配 API Key，不涉及平台计费 |
| NG5 | 不做浏览器历史记录管理 | 范围聚焦书签，历史记录是独立需求 |

## 5. 用户画像

### 画像 A：信息囤积型用户

- **特征**：收藏数百个书签但很少整理，"先收藏再说"心态
- **痛点**：书签太多找不到，想整理但不知从何下手
- **核心需求**：一键自动分类 + 批量清理
- **典型场景**：偶尔打开书签管理器，面对几百个书签感到无从下手，最终放弃

### 画像 B：效率型知识工作者

- **特征**：经常收藏技术文档、研究资料，有分类意识但维护成本高
- **痛点**：记得"收藏过什么内容"但记不住标题，原生搜索找不到
- **核心需求**：AI 语义搜索 + 自然语言查找
- **典型场景**：开发时回忆"之前看过一篇关于 Web Worker 最佳实践的文章"，想快速找到

### 画像 C：重度 AI 用户

- **特征**：已有 OpenAI/Claude API Key，习惯用 AI 辅助日常任务
- **痛点**：现有书签工具不支持 AI，想用 AI 管理书签但没有好工具
- **核心需求**：可配置模型 + AI 对话式书签管理
- **典型场景**：希望通过对话界面直接操作书签——"帮我把所有设计相关的书签移到'设计'文件夹"

## 6. 用户故事

### P0 用户故事

**US-01：悬浮球唤起**
> 作为经常使用书签的用户，我希望在网页边缘看到一个轻量悬浮球，点击它即可展开悬浮面板，或悬停时显示快捷操作按钮，以便我不用打开新标签页就能快速管理书签。

**US-02：AI 对话找书签**
> 作为效率型知识工作者，我希望在悬浮面板的对话框中用自然语言描述我要找的书签内容，AI 帮我找到对应书签，以便我不用记住书签的精确标题。

**US-03：一键收藏 + 自动分类**
> 作为信息囤积型用户，我希望点击一个按钮就能收藏当前页面，AI 自动将书签归入合适的分类，以便我省去手动选择文件夹的麻烦。

**US-04：配置 AI 模型**
> 作为重度 AI 用户，我希望在设置中配置自己的 API Key 和选择模型（GPT-4o / Claude / 本地模型），以便我使用自己偏好的 AI 服务。

**US-05：批量删除书签**
> 作为信息囤积型用户，我希望能多选书签后一键批量删除，并在删除前看到确认提示，以便我快速清理不再需要的书签。

### P1 用户故事

**US-06：AI 一键智能分类全部书签**
> 作为书签混乱的用户，我希望点击一个按钮让 AI 分析所有未分类的书签并自动归类，以便我一次性解决书签混乱问题。

**US-07：失效链接检测**
> 作为有大量书签的用户，我希望扩展能自动检测哪些书签链接已经失效（404 等），以便我清理无效书签。

**US-08：重复书签检测**
> 作为经常收藏页面的用户，我希望扩展能帮我找出重复的书签，以便我合并或删除冗余条目。

**US-09：对话式批量操作**
> 作为重度 AI 用户，我希望在对话框中直接用自然语言操作书签（如"删除所有 Twitter 链接"），AI 帮我执行，以便我不用手动逐个操作。

### P2 用户故事

**US-10：书签导入导出**
> 作为需要备份或迁移的用户，我希望能导入/导出标准 HTML 书签文件，以便我在不同浏览器间迁移。

**US-11：深色模式**
> 作为夜间使用电脑的用户，我希望扩展支持深色模式，以便保护眼睛。

## 7. 功能需求

### 7.1 悬浮球与面板交互（P0）

交互入口参考沉浸式翻译的悬浮球设计：在网页边缘显示一个轻量、可配置、低打扰的悬浮球，用户点击后展开完整面板。悬浮球本身也支持快捷操作（悬停时显示），让用户不用展开面板就能完成最常用的收藏/搜索动作。

#### 7.1.1 悬浮球入口

**描述**：扩展安装后，默认在浏览器窗口右侧边缘显示一个 48px × 48px 的圆形/圆角悬浮球。用户可在设置中关闭、调整位置或设置显示规则。悬浮球作为所有交互的入口，点击后展开完整悬浮面板。

**验收标准**：
- [ ] 扩展安装后，在普通网页右下角/右上角显示悬浮球（默认右侧）
- [ ] 悬浮球尺寸默认 48px × 48px，圆角 12px 或胶囊形，支持主题色/白色两种风格
- [ ] 悬浮球支持拖拽改变垂直位置，位置记忆到 `chrome.storage.local`
- [ ] 快捷键 `Alt+B`（可自定义）可展开/收起悬浮面板
- [ ] 在 Chrome 内置页面（chrome://、chrome-extension://、Chrome Web Store）上不显示悬浮球
- [ ] 支持全局开关：在设置中可一键关闭悬浮球（此时只能通过工具栏图标/快捷键使用）
- [ ] 悬浮球默认透明度 30%（可配置），鼠标悬停时恢复为 100% 不透明

**视觉样式**：
- 默认样式：白色背景 + 主色图标，带 1px 浅灰边框 + 轻微阴影
- 极简样式：仅显示主色图标（适合小屏幕）
- 深色模式：自动切换为深色样式

#### 7.1.2 悬浮球快捷操作（悬停展开）

**描述**：鼠标悬停在悬浮球上时，向页面内侧展开一个迷你操作栏（宽度约 140px），提供最高频的操作按钮，用户无需展开完整面板即可快速使用。

**验收标准**：
- [ ] 鼠标悬停 300ms 后展开迷你操作栏，移开 500ms 后自动收起
- [ ] 操作栏默认包含：收藏当前页、AI 搜索、打开设置
- [ ] 用户可在设置中选择"悬停时不显示操作栏"或"常驻显示操作栏"
- [ ] 点击操作栏中的按钮直接执行对应操作（如点击"收藏"立即收藏当前页并 Toast 提示）
- [ ] 点击操作栏的"展开面板"按钮打开完整悬浮面板
- [ ] 操作栏图标带 Tooltip，显示功能名称

**操作栏显示模式**（设置中可选）：
- **隐藏**（默认）：只有悬浮球本身，点击才展开面板
- **悬停时显示**：鼠标悬停悬浮球时显示操作栏
- **常驻显示**：操作栏始终固定在悬浮球旁

#### 7.1.3 悬浮面板展开与收起

**描述**：点击悬浮球或操作栏中的"展开面板"按钮后，从悬浮球位置附近展开完整悬浮面板。面板可拖拽、可调整大小、可最小化。

**验收标准**：
- [ ] 点击悬浮球展开悬浮面板，再次点击或点击外部区域收起
- [ ] 面板展开时有平滑过渡动画（< 300ms），从悬浮球位置展开或缩放出现
- [ ] 面板默认尺寸 380px × 520px，最小宽度 320px，最小高度 400px
- [ ] 面板可拖拽标题栏移动位置，可拖拽右下角调整大小
- [ ] 面板位置与尺寸记忆到 `chrome.storage.local`，刷新页面后保持
- [ ] 面板支持最小化按钮，最小化后恢复为悬浮球
- [ ] 面板支持 ESC 键快速收起

#### 7.1.4 悬浮球配置规则

**描述**：用户可在扩展设置中灵活配置悬浮球的显示位置、透明度、禁用网站、点击行为等规则，避免悬浮球干扰正常浏览。

**验收标准**：
- [ ] **启用/关闭开关**：设置中提供"启用悬浮球"总开关
- [ ] **固定位置**：可选"右侧"（默认）或"左侧"，悬浮球固定在页面边缘
- [ ] **透明度**：悬浮球收起状态时的透明度，范围 0-100，默认 30；100 为完全透明，0 为完全不透明
- [ ] **操作栏显示方式**：可选"隐藏"、"悬停时显示"（默认）、"常驻显示"
- [ ] **点击行为**：设置点击悬浮球时的默认行为
  - 展开面板（默认）
  - 收藏当前页
  - 打开 AI 对话搜索
- [ ] **禁用网站**：用户可以添加 URL 匹配规则，指定哪些网站不显示悬浮球
  - 支持通配符规则，例如 `*://*.youtube.com/*`、`https://www.example.com/*`
  - 提供默认规则：银行、支付类网站（如 `*://*.alipay.com/*`）默认不显示
  - 列表支持添加、编辑、删除、展开/折叠
  - 提供"当前网站禁用"快捷入口：右键点击悬浮球 → "在 xxx.com 隐藏悬浮球"
- [ ] **白名单模式**（可选）：只在指定网站显示悬浮球，其余网站隐藏
- [ ] **全屏/视频模式自动隐藏**：进入全屏或检测到页面播放视频时，悬浮球自动隐藏

**禁用网站规则示例**：

```
// 默认隐藏规则（内置）
*://*.alipay.com/*
*://*.wechat.com/*
*://*.paypal.com/*
*://*.bankcomm.com/*

// 用户自定义规则
*://*.youtube.com/*
*://*.bilibili.com/*
```

**技术约束**：
- 使用 Content Script 注入页面 DOM，悬浮球和面板均使用 Shadow DOM 隔离样式
- 悬浮球使用 `position: fixed` 固定在视口边缘，不随页面滚动而移动
- 避免在视频全屏、iframe、PDF 页面、编辑器类页面（如 Figma、CodePen）显示悬浮球
- 注入前检测页面 CSP，如 CSP 阻止则降级为 Popup 模式或工具栏图标触发
- 所有配置规则通过 `chrome.storage.local` 同步到各标签页

### 7.2 AI 对话框（P0）

#### 7.2.1 对话式书签搜索

**描述**：悬浮面板内置一个 AI 对话界面。用户输入自然语言查询（如"找一下我收藏的 React Hooks 教程"），AI 基于书签库进行语义检索并返回匹配的书签列表。

**验收标准**：
- [ ] 对话框支持自然语言输入，回车发送
- [ ] AI 返回匹配的书签列表（标题 + URL + 所属分类），最多返回 10 条
- [ ] 用户点击结果中的书签可直接在新标签页打开
- [ ] AI 响应时间 < 5 秒（取决于模型 API 延迟）
- [ ] 对话历史在面板会话内保持，刷新页面后清空
- [ ] AI 未配置时，对话框显示"请先在设置中配置 AI 模型"引导提示
- [ ] 支持流式输出（SSE），用户能看到 AI 正在逐字返回结果

**技术方案**：
- 首次使用时，提取所有书签的标题和 URL 构建索引
- 用户查询时，将书签列表作为 context 注入 AI Prompt，让 AI 进行语义匹配
- 书签数量较大（> 500）时，先做关键词预筛选再送入 AI 做精排

#### 7.2.2 对话式批量操作

**描述**：用户可在对话框中直接发出操作指令，AI 解析意图后执行书签操作（需用户确认）。

**验收标准**：
- [ ] 用户可输入"把所有和 Python 相关的书签移到'编程'文件夹"
- [ ] AI 解析指令后展示操作预览（影响的书签列表 + 目标操作），需用户点击"确认执行"
- [ ] 执行前显示影响范围（"将移动 12 个书签到'编程'文件夹"）
- [ ] 危险操作（删除）需二次确认
- [ ] 操作完成后 AI 回复执行结果摘要

### 7.3 一键收藏 + AI 自动分类（P0）

#### 7.3.1 一键收藏当前页面

**描述**：悬浮面板提供"收藏当前页面"按钮，点击后立即将当前标签页的 URL 和标题保存为书签，同时触发 AI 自动分类。

**验收标准**：
- [ ] 面板上显示醒目的"收藏此页"按钮
- [ ] 点击后 < 1 秒内书签创建完成
- [ ] 书签创建后立即触发 AI 分类（异步，不阻塞用户操作）
- [ ] AI 分类完成后面板内显示 Toast 提示"已收藏到 [分类名称]"
- [ ] 如当前页面已收藏，按钮显示为"已收藏"，点击可取消收藏
- [ ] 收藏时自动抓取页面 meta description 和 favicon 作为补充信息

#### 7.3.2 AI 自动分类

**描述**：收藏书签后，AI 分析页面标题、URL、描述信息，自动将书签归入最合适的分类文件夹。如分类不存在则自动创建。

**验收标准**：
- [ ] AI 分析返回分类建议（分类名 + 置信度）
- [ ] 置信度 > 70% 时自动归入该分类
- [ ] 置信度 < 70% 时放入"待分类"文件夹，并在面板中提示用户手动确认
- [ ] 用户可在设置中关闭自动分类，改为仅收藏不分类
- [ ] 用户可在 Toast 提示中点击"更改"快速调整分类
- [ ] 已有分类文件夹优先复用，避免创建重复分类

**AI Prompt 设计方向**：
```
你是一个书签分类助手。根据以下书签信息，返回最合适的分类。
书签标题: {title}
URL: {url}
描述: {description}
现有分类: {existing_folders}

请从现有分类中选择最合适的，或建议一个新分类名。返回 JSON: {"category": "...", "confidence": 0.0-1.0}
```

### 7.4 模型配置（P0）

#### 7.4.1 AI 模型设置面板

**描述**：设置页提供 AI 模型配置入口，用户可选择模型服务商、填写 API Key、调整参数。

**验收标准**：
- [ ] 支持的模型服务商：OpenAI（GPT-4o / GPT-4o-mini）、Anthropic（Claude 3.5 Sonnet / Haiku）、自定义（OpenAI 兼容 API）
- [ ] API Key 输入框，密码模式显示，支持显示/隐藏切换
- [ ] API Key 存储在 `chrome.storage.local` 中
- [ ] 提供"测试连接"按钮，验证 API Key 有效性

### 9.1 技术选型

### 9.3 关键技术决策

**Q：AI 搜索如何实现语义匹配？**

方案：v1 采用"Prompt 注入"策略——将用户书签列表（标题+URL）按关键词预筛后注入 AI Prompt，让 AI 进行语义匹配。优点是无需额外向量数据库，实现简单。缺点是书签数量大时 Token 消耗高。

v2 规划：引入本地 Embedding（如 `chrome.ml` API 或 TensorFlow.js），对书签做向量化索引，支持离线语义搜索。

**Q：Content Script 注入策略？**

使用 `chrome.scripting.executeScript` 在用户激活时注入，避免在所有页面常驻注入导致性能问题。悬浮面板使用 Shadow DOM 隔离样式。

**Q：API Key 安全性？**

- API Key 存储在 `chrome.storage.local`（仅扩展可访问）
- 不通过任何网络请求传输 API Key（直接从扩展发起 AI API 调用）
- 设置面板中 API Key 默认密码模式显示
