## 0. 为什么要做 v2

v1 已经实现了「AI 对话找书签」「一键收藏 + 自动分类」「悬浮球入口」「批量管理」这些 P0 功能，但用户的真实反馈是 **"很多功能不好用"**。原因不是功能做错了，而是**价值主张停留在 v1 的"管理工具"心智**，而用户真正要的是：

> **"我收藏过的那些页面，不要只是躺在那里安静地积灰，而是要在合适的时刻重新出现在我面前，并形成我自己的知识脉络。"**

v2 的核心命题只有一个：**让书签从"存档"变成"可被重新发现的知识资产"**。围绕这个命题，所有新功能只回答三个问题——

1. **怎么看？** — 提供多种形态的展示，让用户从不同维度重新理解自己的收藏
2. **怎么再发现？** — 让旧书签主动浮现，而不是等用户自己翻文件夹
3. **怎么增值？** — 让孤立的书签产生联系，从 URL 列表变成可思考的知识

## 1. 客户痛点（双视角）

### 1.1 用户视角：我在用 Chrome 书签时真正难受的地方

| # | 痛点 | 真实场景 | 情绪 |
|---|---|---|---|
| P1 | **书签坟场** | 收藏了 300+，真正回头看的 < 30 个，其余永远沉底 | 焦虑、挫败 |
| P2 | **找不回** | "我看过一篇 React 性能优化的文章"——记不住标题，关键词搜不到 | 挫败 |
| P3 | **分类僵化** | 一个书签属于多个主题，但 Chrome 只允许一个文件夹；我每次都得选一个，丢掉其他维度 | 妥协、凑合 |
| P4 | **失效堆积** | 几十个 404、跳转、内容已删的书签我都不知道，列表里全是死链 | 困惑 |
| P5 | **重复冗余** | 同一篇文章收藏了 3 次，混在不同文件夹里 | 烦躁 |
| P6 | **没有上下文** | 半年后回看一个书签，"我当时为啥收藏这个？" 想不起来 | 迷失 |
| P7 | **形式单一** | 永远是文件夹+列表，看不到「时间线」「主题」「关联」 | 无聊、麻木 |
| P8 | **价值感低** | 工具装上几周后就忘了它的存在，只在少数时刻想起来 | 工具浪费 |
| P9 | **跨设备割裂** | 公司电脑收藏的，回到家找不到 | 工作流断裂 |
| P10 | **读不下去** | 收藏了一堆「稍后读」的文章，但永远点不开 | 内疚 |
| P11 | **不会主动整理** | 知道要整理，但每次打开都嫌烦，越攒越多 | 拖延 |
| P12 | **跟笔记工具割裂** | Notion/Obsidian 已经记录了思考，但收藏的页面是断开的 | 知识断层 |

### 1.2 PM 视角：为什么用户付费意愿低、留存差

| # | 诊断 | 现状根因 |
|---|---|---|
| PM1 | **功能被低估** | 装上→偶尔用→卸载，因为用户感知不到"价值时刻" |
| PM2 | **缺乏 Aha Moment** | 没有"惊喜时刻"——第一次用 AI 搜索可能觉得新鲜，但不会有"哇这工具离不开"的感觉 |
| PM3 | **使用频率低** | 仅在收藏/查找时使用，平时不被想起 |
| PM4 | **没有数据沉淀** | 用户没有理由每天打开，工具没有"养熟"的过程 |
| PM5 | **社交/分享缺位** | 用户没动力推荐给朋友——没有可分享的成果 |
| PM6 | **商业化路径模糊** | 单纯"管理书签"价值有限，难以做付费分层 |
| PM7 | **抗替代性弱** | 一旦 Chrome 自带 AI 能力，本扩展核心功能可能被替代 |

## 4. 新形态展示（v2 的「怎么看」）

### 4.1 六种视图（首屏可切换）

```
[对话] [列表] [时间轴] [主题] [知识图谱] [画布]
```

| 视图 | 用途 | 关键体验 | 优先级 |
|---|---|---|---|
| **对话**（已有） | 自然语言查找 | AI 流式回复、可执行操作 | P0 已有 |
| **列表**（已有） | 高密度浏览 | 虚拟滚动、批量操作 | P0 已有 |
| **时间轴** | 看"我什么时候收藏了什么" | 按天/周/月分组、缩略图预览、可看冷热 | **P0 新增** |
| **主题** | 看"我有哪些主题" | AI 自动聚类、主题卡片、显示相关度 | **P1 新增** |
| **知识图谱** | 看"书签之间的关系" | 力导向图、节点=书签、边=共同标签/AI 关联 | **P1 新增** |
| **画布** | 自由组织、像 Notion 一样拼贴 | 拖拽卡片、连箭头、加文字笔记 | **P2 新增** |

### 4.2 视图切换的导航设计

- 顶部 Tab 横滑（移动端友好）
- 键盘快捷键：`Cmd/Ctrl + 1~6` 直达视图
- 视图偏好记忆到 `chrome.storage.local`

### 4.3 各视图详情

#### 4.3.1 时间轴（P0）

**用户价值**：让用户对"我的数字积累"产生具象感知

**关键能力**：
- 按日期分组（今天 / 昨天 / 上周 / 本月 / 今年 / 更早）
- 每个书签卡片显示：favicon、标题、域名、收藏时间、所属分类
- 支持 AI 标注"这是你 N 个月前的收藏，看看是否还感兴趣？"
- 顶部过滤：按分类、按标签、按时间范围

**技术要点**：
- 已有 `chrome.bookmarks.getTree()` 数据，需新增按 `dateAdded` 排序的索引
- 卡片用虚拟滚动（已有 `@tanstack/react-virtual`）
- 与"再发现"机制联动：时间轴视图下，给冷门书签加 ✨ 角标

#### 4.3.2 主题视图（P1）

**用户价值**：让"零散书签"产生结构

**关键能力**：
- AI 自动聚类（embedding + 相似度），无须用户预设
- 主题卡片：标题、代表书签、总数、最近活跃度、置信度
- 点击进入主题详情：按阅读路径排序的书签列表 + AI 生成的"主题摘要"
- 用户可手动合并/拆分主题，AI 学习偏好

**示例**：
```
┌──────────────┬──────────────┬──────────────┐
│ 📚 React 进阶 │ 🎨 设计灵感  │ 📰 AI 资讯    │
│ 12 个书签     │ 8 个书签     │ 23 个书签     │
│ 上次 3 天前   │ 上次 2 周前  │ 上次今天      │
└──────────────┴──────────────┴──────────────┘
```

#### 4.3.3 知识图谱（P1）

**用户价值**：把"看似无关"的书签串起来，发现隐藏关联

**关键能力**：
- 节点 = 书签，边 = 共享标签 / AI 识别的语义相似
- 力导向图布局，可拖拽缩放
- 节点大小 = 收藏频次 / 重要性
- 颜色 = 分类 / 主题
- 悬停节点高亮关联的书签 + 标签列表
- 点击节点展开书签详情侧栏
- 支持聚焦：以某个书签为中心显示其 N 度关联

**技术要点**：
- 使用 Cytoscape.js / react-force-graph
- 关联强度 = jaccard(tag) + embedding cosine
- 性能：超过 200 节点时只渲染活跃子图

#### 4.3.4 画布视图（P2，差异化杀手锏）

**用户价值**：让用户能像 Notion 一样"摆"书签，加自己的思考

**关键能力**：
- 无限画布（米白/深色底）
- 拖入书签生成卡片
- 卡片可连线、加箭头（表达"前置知识"关系）
- 卡片旁可加文字笔记（自动存到本地）
- 支持嵌入图片、引用本地 markdown
- 一键导出为图片 / PDF / 网页（**社交分享入口**）

**与 Notion 的差异**：免登录、纯本地、与浏览器书签双向同步

## 6. 重点功能详设（v2 首发必做）

### 6.1 再发现（Daily Resurfacing）— F3 ⭐

**机制**：每天主动向用户推送 3-5 条"你可能忘了的"老书签

**触发**：
- 时间：用户首次安装满 7 天开始推送
- 时机：每天早上 9:00（可配）、用户首次打开浏览器时
- 数量：每次 3-5 条
- 渠道：浏览器通知 + 面板内"再发现"入口

**选书算法**：
```
score = (
  (now - lastVisited) * 0.3 +   // 越久没看分数越高
  (now - dateAdded) * 0.2 +     // 越老的书签分越高（但不超过 N）
  topic_match(currentBrowsing) * 0.3 +  // 与当前浏览主题相关
  bookmark_quality * 0.2        // 用户评分/访问频次
)
```

**交互**：
- 通知点击 → 直接打开书签
- 面板内 → 显示卡片流，可一键"再收藏"、"归档"、"删除"
- 一键"换一批"

**用户控制**：
- 可关闭推送
- 可调整推送频率（每天 / 每周 / 关闭）
- 可标记"不感兴趣"，AI 学习避免推荐

**Aha 价值**：让用户感觉"这工具懂我"，提升日活

### 6.2 失效链接检测 + 一键清理 — F1

**机制**：
- 后台定期扫描（每天凌晨 + 用户主动触发）
- 标记失效状态：404 / 5xx / 重定向到首页 / 内容明显已删 / DNS 失败
- 在面板"清理"入口展示，按失效时间分组

**交互**：
- 一键全选失效 → 二次确认 → 批量删除
- 支持"保留备份到 Archive.org"选项
- 检测结果导出为 CSV

**隐私**：仅访问用户书签本身，不上传到任何服务器（架构约束）

### 6.3 多标签系统 — F5

**设计**：
- 标签独立于文件夹，文件夹保留作为"主要归档位置"
- 一个书签可打多个标签
- 标签支持层级（如 `tech/frontend/react`）
- 标签视图可替代文件夹视图

**迁移策略**：
- 首次启用时，把现有文件夹名作为初始标签
- 用户可选择保留文件夹作为主路径

**与 AI 结合**：
- AI 分类时同时建议标签
- 标签可作为知识图谱的边权重

### 6.4 主题视图 — F7

**AI 聚类方案**：
1. 抽取每个书签的 embedding（用配置的模型生成向量，存本地）
2. 用 HDBSCAN / k-means 聚类
3. AI 为主题命名（用 LLM 生成人类可读的主题名）
4. 增量更新：新书签入队 → 异步重新计算归属

**离线友好**：
- 支持纯本地 embedding（用本地模型如 all-MiniLM）
- 缓存聚类结果，避免重复计算

**交互**：
- 主题卡片显示：名称、代表书签、置信度、活跃度
- 进入主题显示该主题的所有书签 + AI 生成摘要
- 用户可手动调整主题边界

## 7. 用户旅程（v2 优化版）

### 7.1 首次使用（Day 0）

```
安装 → 进入设置 → 配置 AI → 回到任意网页 → 看到悬浮球 → 收藏第一个书签
                                            ↓
                                  看到 AI 自动归类 + 建议标签
                                            ↓
                                  体验"原来这就是 AI 书签管家"
```

**关键优化**：在用户第一次收藏后，立即展示"v2 的 3 个新能力预告"，引导用户探索。

### 7.2 留存关键期（Day 1-7）

```
Day 1:  体验 AI 搜索（核心卖点）
Day 2:  收到第一次"再发现"推送
Day 3:  看到时间轴视图，理解"我的积累"
Day 7:  触发失效链接检测，体验一键清理 → 形成"工具离不开"的认知
```

**Aha 时刻顺序**：AI 搜索 → 再发现 → 失效清理

### 7.3 长期使用（Day 30+）

```
每日: 打开面板看"今日再发现" → 浏览 1-2 条
每周: 进入时间轴 → 回顾本周收藏 → 进入主题视图 → 看主题变化
每月: 进入知识图谱 → 发现新关联 → 进入画布 → 整理一个主题
每季: 数据导出/同步备份
```

## 8. 成功指标

### 8.1 北极星指标

### 8.3 反指标（不能为了短期牺牲长期）

- 单次 AI 调用成本必须 < ¥0.05（用户自配 API Key 时不感知）
- 单次推送不能让用户觉得烦（设阈值，连续 3 次不打开就降频）
- 性能：1000 书签的视图切换 < 200ms

---

## 9. 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| Chrome 内置 AI 替代核心功能 | 高 | 加速走"再发现 + 主题视图"差异化路线 |
| 用户不愿付费 | 中 | 云同步 + 主题订阅作为付费钩子，基础功能保持免费 |
| AI 成本失控 | 中 | 默认用本地 embedding + 缓存，AI 调用仅在"创造价值时刻" |
| 知识图谱性能差 | 中 | 子图渲染 + 懒加载 + 虚拟化 |
| 标签系统与文件夹冲突 | 中 | 提供"标签为主 / 文件夹为主 / 并存"三种模式 |

## 11. 一句话总结

> **v2 不是做"更多功能"，而是让书签从"被管理的对象"变成"会主动服务用户的知识伙伴"。**
> 
> **核心是把"找书签"的负担从用户转移到产品——我们不再等用户来搜，而是把对的书签在对的时间送到用户面前。**
